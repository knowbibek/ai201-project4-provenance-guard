# Provenance Guard — Planning & Spec

> Design document, written **before** implementation (Milestone 2). This is also the primary
> prompting tool for Milestones 3–5: each implementation milestone is given the relevant sections
> below plus the architecture diagram. Specific answers here → specific code there.
>
> Reference architecture: CodePath's **RepairSafe** lab (LLM-as-judge classifier + `.jsonl` audit
> log). We reuse its *prompt → parse → validate → fail-closed fallback* shape for Signal 1 and its
> structured-logging discipline, but build on **Flask** with a **SQLite** store (mutable appeal status).

---

## Problem & Goals

Creative platforms need to tell readers whether a piece was likely written by a human or generated by
AI — **to protect attribution and give context, not to police creativity.** Provenance Guard is a
pluggable backend that (1) classifies submitted text, (2) scores confidence honestly instead of forcing
a binary, (3) surfaces a plain-language transparency label, and (4) lets creators appeal.

**Core design bias — false positives are the worst outcome.** Wrongly labeling a human's work as AI
does real reputational harm; missing some AI is a softer failure. Every tradeoff tilts *away* from
accusing a human: a higher evidence bar to say "AI," an honest "uncertain" whenever signals disagree,
and — borrowed from RepairSafe — a **fail-closed fallback**: when anything is unparseable or unreliable,
we fall back toward *uncertain*, never toward an AI accusation.

---

## 1. Detection Signals

Two **distinct** signals — one semantic, one structural — combined into a single confidence score.
Each signal outputs an **AI-likelihood in [0,1]** (0 = clearly human, 1 = clearly AI) plus metadata.

### Signal 1 — LLM semantic classifier (Groq `llama-3.3-70b-versatile`)

- **Measures:** holistic voice/coherence — does the text *read* as human? Catches generic phrasing,
  even hedging, over-smooth transitions, absence of idiosyncratic voice.
- **Implementation shape (from RepairSafe):** build prompt → single chat completion (no tools, no
  history, `temperature=0` for stability) → parse → validate → fail-closed fallback.
- **Prompt asks for strict JSON:**
  ```json
  {"ai_likelihood": 0-100, "reasoning": "one sentence"}
  ```
- **Output object:**
  ```json
  {"name": "llm", "score": 0.0-1.0, "reasoning": "...", "available": true}
  ```
  `score = ai_likelihood / 100`. Validate `0 ≤ score ≤ 1`.
- **Fail-closed fallback:** if the response can't be parsed, isn't valid JSON, or is out of range →
  `{"name":"llm","score":0.5,"reasoning":"unparseable — treated as uncertain","available":false}`.
  We never fall back to a high AI score.
- **Blind spots:** non-deterministic; weak on very short text; can flag *bland-but-human* writing
  (student/corporate prose) → false-positive risk; genre-sensitive; black box.

### Signal 2 — Stylometric heuristics (pure Python, no external libs)

Three sub-metrics, each mapped to its own AI-likelihood ∈ [0,1] via a clamped linear map, then combined.

| Sub-metric | How computed | Direction | Sub-weight |
|---|---|---|---|
| **Sentence-length burstiness** | split on `.!?`; words per sentence; `CV = stdev/mean` | low CV (uniform) → AI. `CV ≥ 0.60` → 0.0 (human); `CV ≤ 0.30` → 1.0 (AI); linear between | **0.50** |
| **Lexical diversity (MATTR)** | moving-average type–token ratio, window = 50 tokens (length-robust) | very middling diversity → AI-leaning. `MATTR ≥ 0.75` → 0.0; `MATTR ≤ 0.55` → 1.0; linear between | **0.25** |
| **Punctuation variety** | count of *distinct* "expressive" marks used (`— ; : ( ) ?`) per 100 words | sparse/uniform → AI-leaning. presence of varied marks → human | **0.25** |

- **Combined:** `s2 = 0.50·burstiness + 0.25·mattr + 0.25·punct`, clamped to [0,1].
- **Output object:**
  ```json
  {"name":"stylometry","score":0.0-1.0,"metrics":{"cv":..,"mattr":..,"punct":..},
   "available":true,"low_reliability":false}
  ```
- **Length guard (fail-closed):** if text has `< 40 words` or `< 3 sentences`, stats are unstable →
  set `score = 0.5`, `low_reliability = true`. (Drives the short-text edge case below.)
- **Blind spots:** understands no meaning; unstable on short text; broken by poetry/lists/dialogue;
  trivially gamed (a human who writes uniformly, or AI told to "vary sentence length").

### Combining the two → one score

```
w_llm = 0.60, w_sty = 0.40                      # LLM weighted higher: more robust signal
if stylometry.low_reliability:  w_llm, w_sty = 0.85, 0.15   # lean on the reliable signal, cap confidence
combined_ai_likelihood = w_llm·s1 + w_sty·s2
```
**Why 0.6/0.4:** semantic judgment is the stronger evidence; stylometry is corroborating. Weights and
all thresholds are revisited against the labeled test set in M4 (see *AI Tool Plan*).

## 2. Uncertainty Representation

We report two numbers internally and **never show a raw decimal to end users** (you can't meaningfully
explain "0.62" to a reader — we show a confidence *band* + plain language instead).

- **`ai_likelihood ∈ [0,1]`** — direction only (<0.5 human, >0.5 AI).
- **`confidence ∈ [0,1]`** — how strongly *and consistently* the evidence points to the verdict:
  ```
  decisiveness = 2 · |combined − 0.5|        # 0 at the fence, 1 at the extremes
  agreement    = 1 − |s1 − s2|               # 1 when both signals agree, 0 when opposite
  confidence   = decisiveness · (0.4 + 0.6 · agreement)   # disagreement suppresses confidence
  if stylometry.low_reliability: confidence = min(confidence, 0.55)   # short text can't be "high conf"
  ```

**What a confidence of 0.6 means to this system:** *moderately sure.* We treat **≥ 0.50 as enough to
state a "likely human" verdict, but require ≥ 0.65 to state a "likely AI" verdict.** So a 0.6 lands as
**"likely human"** on the human side, but stays **"uncertain"** on the AI side — deliberately, because a
wrong AI call is the costly error. This asymmetry *is* the uncertainty model.

### Thresholds (the three-way map — not a binary flip at 0.5)

| Condition | Verdict / label |
|---|---|
| `confidence < 0.50` | **Uncertain** (regardless of direction) |
| `confidence ≥ 0.50` **and** `ai_likelihood < 0.5` | **Likely human** |
| `0.50 ≤ confidence < 0.65` **and** `ai_likelihood ≥ 0.5` | **Uncertain** (not sure enough to accuse) |
| `confidence ≥ 0.65` **and** `ai_likelihood ≥ 0.5` | **Likely AI** |

A 0.51-confidence result is Uncertain; a 0.95-confidence result is a firm verdict — meaningfully
different outputs, as required. **Calibration:** these scores are *evidence scores*, not true
probabilities. We validate them in M4 on a labeled mini-set (clear-human, clear-AI, ambiguous) and tune
the cutoffs so clear cases clear the bars and ambiguous/short cases fall into Uncertain.

## 3. Transparency Labels (FINAL — canonical copy mirrored to README)

Reader-facing, plain language, cautious by design. Even the AI label is an *estimate*, never an
accusation, and always names the appeal path.

| Variant | Selected when | **Final label text** |
|---|---|---|
| **High-confidence human** | `confidence ≥ 0.50` & human side | ✍️ **Likely written by a person.** Our automated check found strong signs of human authorship. This is an automated estimate, not a guarantee. *(Confidence: High)* |
| **High-confidence AI** | `confidence ≥ 0.65` & AI side | 🤖 **Likely AI-generated.** Our automated check found strong signs this was produced with AI assistance. This is an estimate, not a certainty — if you wrote this yourself, you can appeal. *(Confidence: High)* |
| **Uncertain** | everything else (incl. disagreement / short text) | ❓ **Origin unclear.** Our automated check couldn't reliably tell whether a person or AI wrote this, so we're not making a claim either way. *(Confidence: Low)* |

## 4. Appeals Workflow

- **Who can appeal:** the creator of a submission (any caller holding the `submission_id`). No auth in
  this prototype — the `submission_id` acts as the claim ticket.
- **What they provide:** `submission_id` + free-text `reasoning` (their case, e.g. "this is my original
  free verse, here's my draft history").
- **What the system does on receipt:**
  1. Look up the original decision by `submission_id` (404 if unknown).
  2. Insert an **appeal row** into the audit log, linked to that decision, storing the reasoning + UTC
     timestamp.
  3. Update the decision's `status` from `classified` → **`under_review`**.
  4. Return `{ appeal_id, submission_id, status: "under_review", logged_at }`.
- **No automated re-classification** — the appeal flags it for a human; that's the intent.
- **What a human reviewer sees in the appeal queue** (`GET /log?type=appeal`, or `GET /appeals`):
  the original text excerpt, the original verdict + both signal scores + confidence, the label shown,
  the creator's reasoning, and timestamps for both the decision and the appeal — everything needed to
  judge the appeal without re-running anything.

## 5. Anticipated Edge Cases

Specific scenarios this system handles poorly, and how the design responds:

1. **Minimalist / repetitive poetry (haiku, spare free verse).** Short, uniform lines → low burstiness
   *and* low lexical diversity → stylometry scores it strongly AI. **Response:** the length guard often
   trips (short), and the LLM usually reads genuine craft → signals disagree → confidence suppressed →
   **Uncertain**, not an AI accusation. This is the canonical false-positive case (full trace below).
2. **Very short submissions (a 2-sentence microfic, a tweet-length blurb).** Both signals are
   unreliable — stylometry stats are unstable, the LLM has little to judge. **Response:** length guard
   sets `low_reliability`, which caps confidence at 0.55 → forced toward **Uncertain**.
3. **Heavily AI-edited human writing (Grammarly/"polish this" hybrids).** Genuinely ambiguous origin —
   part human, part machine. **Response:** signals tend to split → low confidence → **Uncertain**, which
   is the honest answer; there is no correct binary here.
4. **Adversarial AI ("write with varied sentence lengths and a typo or two").** Defeats stylometry by
   construction. **Response:** we accept this blind spot honestly — the LLM may still catch it, but if
   both miss it the result is a false negative (the *acceptable* error direction), never a false human-
   accusation. Documented, not pretended away.

### False-positive walkthrough (the scenario that must work)

A poet submits spare, minimalist free verse. (1) Stylometry sees low CV + low MATTR → leans AI (~0.7).
(2) The LLM reads real craft/voice → leans human (~0.25). (3) Strong disagreement → `agreement` low →
**confidence suppressed**; the AI bar (≥0.65) isn't met → verdict = **Uncertain**. (4) Reader sees the
"Origin unclear … not making a claim" label — *not* an accusation. (5) If the poet still objects, they
`POST /appeal`; status flips to `under_review`, the appeal is logged beside the original decision, and a
human follows up.

## Architecture

A submission enters at `POST /submit`, passes the **rate limiter**, runs through **Signal 1 (LLM)** and
**Signal 2 (stylometry)** in the **detection pipeline**, is fused by **confidence scoring** (weighting +
agreement + asymmetric thresholds) into a verdict, gets a **transparency label**, is written to the
**SQLite audit log**, and returns as JSON. An appeal enters at `POST /appeal`, looks up the original
decision, writes a linked appeal row, flips the decision's status to `under_review`, and returns
confirmation. `GET /log` is a read-only view over the audit log.

### Flow 1 — Submission

```

   raw text                 raw text              │
 ┌──────────┐   POST /submit   ┌───────────────────────────┐
 │ Platform │ ───────────────► │       Rate Limiter        │
 │ / client │                  │      (Flask-Limiter)      │
 └──────────┘                  └─────────────┬─────────────┘
       ▲                             raw text │
       │                                       ▼
       │                         ┌───────────────────────────┐
       │                         │      Detection Pipeline    │
       │                         │                            │
       │                         │  raw text ─► Signal 1: LLM  │  ── s1 score (0–1) ─┐
       │                         │              (Groq)         │                     │
       │                         │  raw text ─► Signal 2:      │  ── s2 score (0–1) ─┤
       │                         │              Stylometry     │                     │
       │                         └─────────────────────────────┘                    │
       │                                                       two signal scores      ▼
       │                                                   ┌───────────────────────────────┐
       │                                                   │       Confidence Scoring        │
       │                                                   │  weighting + agreement +        │
       │                                                   │  asymmetric thresholds          │
```

## API Surface (the contract)

| Method | Path | Accepts | Returns |
|---|---|---|---|
| POST | `/submit` | `{ content: str, creator_id?: str, title?: str }` | `{ submission_id, verdict, ai_likelihood, confidence, label:{variant,text}, signals:[{name,score,...}], created_at }` |
| POST | `/appeal` | `{ submission_id: str, reasoning: str }` | `{ appeal_id, submission_id, status:"under_review", logged_at }` |
| GET | `/log` | `?type=decision\|appeal&limit=` (optional) | `[ {id, type, ...} ]` newest first |
| GET | `/submission/<id>` | — (optional convenience) | the stored decision |
| GET | `/health` | — (optional) | `{ status: "ok" }` |

**Required:** `/submit`, `/appeal`, `/log`.

---

## Data Model (SQLite)

- **`decisions`**: `id` (uuid), `created_at` (UTC ISO `…Z`), `creator_id`, `title`, `content_hash`
  (sha256 — we store a hash + excerpt, not necessarily full text), `content_excerpt` (first 300 chars),
  `verdict`, `ai_likelihood`, `confidence`, `s1_llm_score`, `s2_stylometry_score`, `signals_json`
  (full per-signal detail), `label_variant`, `label_text`, `status` (`classified` | `under_review`).
- **`appeals`**: `id` (uuid), `submission_id` (→ `decisions.id`), `reasoning`, `created_at`.

`GET /log` unions both tables, newest first. On every decision and appeal we also print a one-line
console summary (RepairSafe style), e.g. `[DECISION] verdict=uncertain conf=0.41 "Spare free verse…"`.

## Rate Limiting (limits chosen + reasoning)

Keyed per client IP (`flask_limiter.util.get_remote_address`); in-memory store for the prototype
(`memory://` — noted as single-process only; production would use Redis).

| Endpoint | Limit | Reasoning |
|---|---|---|
| `POST /submit` | **10 / minute, 100 / hour, 300 / day** | Each call costs a Groq request. A real creator submits a handful of pieces per session; 10/min absorbs honest bursts (re-checking edits) without friction, while the hourly/daily caps blunt a script trying to flood the paid LLM endpoint. |
| `POST /appeal` | **5 / minute, 20 / hour** | Appeals are rare and human-initiated; low caps stop appeal-spam against the review queue while never blocking a legitimate creator. |
| `GET /log` | **30 / minute** | Read-only and cheap, but still capped to discourage scraping the audit trail. |

Over-limit → HTTP `429` with a JSON error; the detection pipeline never runs, protecting Groq spend.

## AI Tool Plan (Milestones 3–5)

For each milestone: which spec sections to hand the AI tool, what to ask it to generate, how to verify.

### M3 — Submission endpoint + Signal 1
- **Provide:** §1 *Detection Signals* (Signal 1 subsection), the *Architecture* diagram, *API Surface*
  (`/submit` row), *Data Model* (`decisions`), and the RepairSafe `config.py`/`safety.py` shape.
- **Ask for:** a Flask app skeleton with `POST /submit`, a `config.py`, and `signal_llm(text) -> dict`
  implementing the prompt → parse → validate → **fail-closed** pattern. Wire `/submit` to call Signal 1
  only for now and return its raw score.
- **Verify:** call `signal_llm()` directly on ~4 texts (2 obviously human, 2 obviously AI) and confirm
  scores point the right way and that malformed/empty input hits the 0.5 fallback. Then curl `/submit`.

### M4 — Signal 2 + confidence scoring
- **Provide:** §1 (Signal 2 + combining), §2 *Uncertainty Representation* (formulas + threshold table),
  the diagram.
- **Ask for:** `signal_stylometry(text) -> dict` (the three sub-metrics + length guard) and a
  `score_confidence(s1, s2) -> {ai_likelihood, confidence, verdict}` implementing the fusion +
  agreement + asymmetric thresholds; integrate both into `/submit`.
- **Verify:** run a labeled mini-set (clear-human, clear-AI, ambiguous, short, a minimalist poem) and
  check: clear cases get high confidence + correct direction; ambiguous/short/poem cases fall into
  **Uncertain**; nudging one signal moves the combined score the expected way. Tune weights/cutoffs here.

### M5 — Production layer (labels, appeals, rate limiting, audit log)
- **Provide:** §3 *Transparency Labels* (final text + selection table), §4 *Appeals*, *Rate Limiting*,
  *Data Model*, the diagram.
- **Ask for:** `build_label(verdict, confidence) -> {variant, text}`; the SQLite audit log
  (`decisions` + `appeals`, console summary) wired into `/submit`; `POST /appeal` (lookup → log →
  status `under_review`); `GET /log`; and Flask-Limiter decorators with the documented limits.
- **Verify:** craft inputs that reach all three label variants; confirm an appeal flips status to
  `under_review` and appears in `GET /log`; confirm `/log` shows ≥3 entries; confirm exceeding the
  `/submit` limit returns `429` without calling Groq.
