# Provenance Guard

A backend service any creative-sharing platform can plug into to classify whether submitted
text was likely written by a human or generated by AI ÔÇö then surface an honest, plain-language
transparency label and let creators appeal a classification.

> **Status:** Milestone 5 ÔÇö production layer complete. Two-signal `POST /submit` with calibrated
> confidence and a confidence-varying transparency label, `POST /appeal` (status Ôćĺ `under_review`),
> Flask-Limiter rate limiting, and a structured appeal-aware audit log. Run `python selftest.py` for
> the calibration check. Full design, diagrams, and the AI Tool Plan live in [planning.md](planning.md).

## Architecture

A submission flows: `POST /submit` Ôćĺ **rate limiter** Ôćĺ **Signal 1 (LLM, Groq)** + **Signal 2
(stylometry)** Ôćĺ **confidence fusion** (weighting + agreement + asymmetric thresholds) Ôćĺ **label
builder** Ôćĺ **audit log (SQLite)** Ôćĺ JSON response. An appeal flows: `POST /appeal` Ôćĺ look up the
original decision Ôćĺ write a linked appeal row + set status `under_review` Ôćĺ response.

Full narrative and labeled diagrams for both flows: [planning.md](planning.md#architecture-diagram).

## Detection Signals

Two **distinct** signals ÔÇö one semantic, one structural ÔÇö so the combination beats either alone:

- **Signal 1 ÔÇö LLM semantic classification (Groq `llama-3.3-70b-versatile`):** judges holistic
  voice/coherence (human vs AI). *Blind spot:* non-deterministic, weak on short text, and can
  flag bland-but-human writing ÔÇö a false-positive risk.
- **Signal 2 ÔÇö Stylometric heuristics (pure Python):** sentence-length variance ("burstiness"),
  typeÔÇôtoken ratio, punctuation density. *Blind spot:* understands no meaning, unstable on short
  text, broken by poetry/lists, and easily gamed.

Their blind spots barely overlap, so disagreement is itself useful signal. See
[planning.md](planning.md#detection-signals).

## Confidence Scoring

We track `ai_likelihood Ôłł [0,1]` (direction: 0 = human, 1 = AI) and a separate `confidence Ôłł [0,1]`.
`ai_likelihood` is the weighted average `0.6┬ĚLLM + 0.4┬Ěstylometry`. Confidence treats the **LLM as the
primary signal** (it's far stronger at this task): `base = 2┬Ě|LLMÔłĺ0.5|` sets the ceiling, and
stylometry then **corroborates** (same side Ôćĺ raises confidence) or **dissents** (opposite side Ôćĺ
erodes it). Because `base` comes only from the LLM, stylometry can never assert a verdict alone ÔÇö if
the LLM is unavailable, confidence collapses to ~0 Ôćĺ `uncertain` (**fail-closed**). Users never see the
raw decimal; they see a confidence *band* + plain language.

The mapping is **not** a binary flip at 0.5 ÔÇö it's an **asymmetric three-way** split (harder to assert
"AI" than "human", because false positives are the worst outcome):

| Condition | Verdict |
|---|---|
| `confidence < 0.50` | **Uncertain** |
| `confidence Ôëą 0.50` & human side | **Likely human** |
| `0.50 ÔëĄ confidence < 0.65` & AI side | **Uncertain** (not sure enough to accuse) |
| `confidence Ôëą 0.65` & AI side | **Likely AI** |

So a confidence of **0.6 means "moderately sure"**: enough to state *likely human*, but on the AI side
it stays *uncertain*. Full formulas + calibration plan: [planning.md](planning.md#2-uncertainty-representation).

**How we tested it's meaningful:** run `python selftest.py` for the four-input calibration check. On
real signal scores it produces: **clear-AI Ôćĺ `likely_ai`** (conf Ôëł 0.76), **clear-human Ôćĺ
`likely_human`** (conf Ôëł 0.84), **formal-human Ôćĺ `uncertain`** (false positive avoided), **edited-AI Ôćĺ
`likely_human`**. The fusion in [scoring.py](scoring.py) was additionally checked against seven
boundary pairs, all matching spec, including the two that matter most: opposite signals (the poem) Ôćĺ
`uncertain`, and **LLM-unavailable + stylometry screaming AI Ôćĺ `uncertain`** (fail-closed ÔÇö stylometry
can't accuse on its own). All three verdicts are reachable and the asymmetric AI bar holds.

## Transparency Labels

The reader-facing label has three variants (final text below). Even the AI label is phrased as an
*estimate*, never an accusation, and names the appeal path.

| Variant                | Label text |
|------------------------|------------|
| High-confidence human  | ÔťŹ´ŞĆ **Likely written by a person.** Our automated check found strong signs of human authorship. This is an automated estimate, not a guarantee. *(Confidence: High)* |
| High-confidence AI     | ­čĄľ **Likely AI-generated.** Our automated check found strong signs this was produced with AI assistance. This is an estimate, not a certainty ÔÇö if you wrote this yourself, you can appeal. *(Confidence: High)* |
| Uncertain              | ÔŁô **Origin unclear.** Our automated check couldn't reliably tell whether a person or AI wrote this, so we're not making a claim either way. *(Confidence: Low)* |

## Appeals

A creator who disputes a classification calls `POST /appeal` with a `content_id` and free-text
`creator_reasoning`. The system looks up the original decision, writes a linked row to the `appeals`
table, flips the decision's status to **`under_review`** (and denormalizes the reasoning onto the
decision so one `GET /log` entry shows everything), and returns confirmation. No automated
re-classification ÔÇö appeals are flagged for a human reviewer, who sees the original text, verdict,
signal scores, confidence, label, and the creator's reasoning. Confirmation response:

```json
{"appeal_id": "5f2bc4d1-ÔÇŽ", "content_id": "b9a9116a-ÔÇŽ", "status": "under_review",
 "logged_at": "2026-06-28T18:28:43.969Z", "message": "Appeal received. This content is now under review."}
```
See [planning.md](planning.md#4-appeals-workflow).

## Rate Limiting

Per client IP, via Flask-Limiter. Limits chosen for a realistic writing platform and to protect the
(cost-bearing) Groq calls from flooding:

| Endpoint | Limit | Why |
|---|---|---|
| `POST /submit` | **10/min, 100/hour, 300/day** | Each call costs a Groq request. A real creator submits a handful per session; 10/min absorbs honest re-check bursts, while hourly/daily caps blunt scripted flooding of the paid endpoint. |
| `POST /appeal` | **5/min, 20/hour** | Appeals are rare and human-initiated; low caps stop appeal-spam without blocking a legitimate creator. |
| `GET /log` | **30/min** | Read-only and cheap, but capped to discourage scraping the audit trail. |

Over-limit Ôćĺ HTTP `429` with a JSON error; the detection pipeline never runs.

**Evidence** ÔÇö 12 rapid `POST /submit` requests against the 10/min limit (first 10 pass, rest blocked):

## Audit Log

Every `/submit` writes one structured row to a SQLite store (`provenance.db`, table `decisions`);
every `/appeal` writes an `appeals` row and flips the decision's `status`. Both print a one-line
console summary. `GET /log` returns the entries as JSON, newest first. Each row records: `content_id`,
`creator_id`, `title`, `timestamp` (UTC ISO `ÔÇŽZ`), `content_excerpt`, `attribution`, `confidence`,
`ai_likelihood`, `llm_score`, `stylometry_score`, `label_variant`/`label_text`, the full `signals`
array, `status`, and (if appealed) `appeal_id` / `appeal_reasoning` / `appealed_at`.

Example `GET /log` output ÔÇö 3 structured entries, **one appealed** (`status: under_review` with
`appeal_reasoning`). LLM scores read `0.5`/`uncertain` only because the offline build environment
couldn't reach Groq; run locally for real scores:
