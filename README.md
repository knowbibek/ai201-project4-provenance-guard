# Provenance Guard

A backend service any creative-sharing platform can plug into to classify whether submitted
text was likely written by a human or generated by AI — then surface an honest, plain-language
transparency label and let creators appeal a classification.

> **Status:** Submission-ready (M6). All seven required features implemented and verified end-to-end:
> two-signal `POST /submit` with calibrated, confidence-varying transparency labels, `POST /appeal`
> (status → `under_review`), Flask-Limiter rate limiting, and a structured appeal-aware audit log. Run
> `python selftest.py` for the calibration check. Full design, diagrams, and the AI Tool Plan live in
> [planning.md](planning.md).

**Design thesis:** AI detection is an unsolved problem, so this system is built to *communicate
uncertainty honestly* rather than fake certainty. Its single strongest bias: **a false positive
(calling a human's work AI) is the worst outcome**, so it is harder to assert "AI" than "human," and
genuine ambiguity resolves to "uncertain" with a path to appeal.

## Architecture

A submission flows: `POST /submit` → **rate limiter** → **Signal 1 (LLM, Groq)** + **Signal 2
(stylometry)** → **confidence fusion** (weighting + agreement + asymmetric thresholds) → **label
builder** → **audit log (SQLite)** → JSON response. An appeal flows: `POST /appeal` → look up the
original decision → write a linked appeal row + set status `under_review` → response.

Full narrative and labeled diagrams for both flows: [planning.md](planning.md#architecture-diagram).

## Detection Signals

Two **distinct** signals — one semantic, one structural — so the combination beats either alone:

- **Signal 1 — LLM semantic classification (Groq `llama-3.3-70b-versatile`):** judges holistic
  voice/coherence (human vs AI). *Blind spot:* non-deterministic, weak on short text, and can
  flag bland-but-human writing — a false-positive risk.
- **Signal 2 — Stylometric heuristics (pure Python):** sentence-length variance ("burstiness"),
  type–token ratio (MATTR), punctuation variety. *Blind spot:* understands no meaning, unstable on
  short text, broken by poetry/lists, and easily gamed.

**Why these two (the reasoning, not just the what):** the requirement is *independence* — two signals
that fail in different ways. The LLM reasons about *meaning and voice*; stylometry measures *form* and
never reads a word for sense. Their blind spots barely overlap, which is the point: when a polished AI
paragraph fools stylometry, the LLM still catches it; when the LLM over-flags formal human prose,
stylometry's disagreement pulls the verdict back toward "uncertain." A second LLM prompt would have
been easier but *correlated* — it would share the first one's blind spots and add little. Stylometry is
deliberately the cheap, transparent, offline counterweight to the opaque, paid, non-deterministic LLM.

**What I'd change for a real deployment:** (1) replace the hand-tuned stylometric cutoffs with values
calibrated on a labeled corpus per genre (poetry vs. essays need different burstiness baselines);
(2) run the LLM signal at low temperature across a few samples and average, to tame its
non-determinism; (3) add a third, independent signal (e.g. a perplexity/detector model) so a single
signal's failure can be outvoted rather than just lowering confidence. See
[planning.md](planning.md#detection-signals).

## Confidence Scoring

We track `ai_likelihood ∈ [0,1]` (direction: 0 = human, 1 = AI) and a separate `confidence ∈ [0,1]`.
`ai_likelihood` is the weighted average `0.6·LLM + 0.4·stylometry`. Confidence treats the **LLM as the
primary signal** (it's far stronger at this task): `base = 2·|LLM−0.5|` sets the ceiling, and
stylometry then **corroborates** (same side → raises confidence) or **dissents** (opposite side →
erodes it). Because `base` comes only from the LLM, stylometry can never assert a verdict alone — if
the LLM is unavailable, confidence collapses to ~0 → `uncertain` (**fail-closed**). Users never see the
raw decimal; they see a confidence *band* + plain language.

The mapping is **not** a binary flip at 0.5 — it's an **asymmetric three-way** split (harder to assert
"AI" than "human", because false positives are the worst outcome):

| Condition | Verdict |
|---|---|
| `confidence < 0.50` | **Uncertain** |
| `confidence ≥ 0.50` & human side | **Likely human** |
| `0.50 ≤ confidence < 0.65` & AI side | **Uncertain** (not sure enough to accuse) |
| `confidence ≥ 0.65` & AI side | **Likely AI** |

So a confidence of **0.6 means "moderately sure"**: enough to state *likely human*, but on the AI side
it stays *uncertain*. Full formulas + calibration plan: [planning.md](planning.md#2-uncertainty-representation).

**Why this approach:** averaging two likelihoods would let a confidently-correct LLM be silently
dragged to "uncertain" by a stylometry signal that simply *can't tell* (it sits near 0.5 on polished
text). Treating the LLM as the confidence *driver* and stylometry as corroboration/dissent fixes that
while preserving the false-positive guard — stylometry that *actively disagrees* still erodes
confidence to "uncertain."

**Two example submissions (real scores from live runs):**

| | Text | LLM (s1) | Stylometry (s2) | ai_likelihood | confidence | verdict |
|---|---|---|---|---|---|---|
| **High confidence** | *"ok so i finally tried that new ramen place downtown and honestly? underwhelming…"* | 0.20 | 0.167 | **0.187** | **0.84** | `likely_human` |
| **Lower confidence** | *"The relationship between monetary policy and asset price inflation has been extensively studied in the literature…"* | 0.80 | 0.562 | 0.696 | **0.649** | `uncertain` |

The casual human note scores **0.84** confidence — both signals agree it's human, so the system states
it plainly. The formal academic passage is the interesting one: the LLM leans AI (0.80, fooled by the
uniform academic register) and stylometry mildly agrees, yet confidence lands at **0.649 — just below
the 0.65 AI bar** — so the system says **`uncertain`** rather than falsely accusing a human author. That
margin is the false-positive guard doing its job. Same pipeline, **0.84 vs 0.649**, two different
labels: meaningful variation, not a constant.

**How else it was tested:** the fusion in [scoring.py](scoring.py) was checked against seven boundary
pairs, all matching spec, including the two that matter most: opposite signals (the poem case) →
`uncertain`, and **LLM-unavailable + stylometry screaming AI → `uncertain`** (fail-closed — stylometry
can't accuse on its own). All three verdicts are reachable and the asymmetric AI bar holds.

## Transparency Labels

The reader-facing label has three variants (final text below). Even the AI label is phrased as an
*estimate*, never an accusation, and names the appeal path.

| Variant                | Label text |
|------------------------|------------|
| High-confidence human  | ✍️ **Likely written by a person.** Our automated check found strong signs of human authorship. This is an automated estimate, not a guarantee. *(Confidence: High)* |
| High-confidence AI     | 🤖 **Likely AI-generated.** Our automated check found strong signs this was produced with AI assistance. This is an estimate, not a certainty — if you wrote this yourself, you can appeal. *(Confidence: High)* |
| Uncertain              | ❓ **Origin unclear.** Our automated check couldn't reliably tell whether a person or AI wrote this, so we're not making a claim either way. *(Confidence: Low)* |

## Appeals

A creator who disputes a classification calls `POST /appeal` with a `content_id` and free-text
`creator_reasoning`. The system looks up the original decision, writes a linked row to the `appeals`
table, flips the decision's status to **`under_review`** (and denormalizes the reasoning onto the
decision so one `GET /log` entry shows everything), and returns confirmation. No automated
re-classification — appeals are flagged for a human reviewer, who sees the original text, verdict,
signal scores, confidence, label, and the creator's reasoning. Confirmation response:

```json
{"appeal_id": "5f2bc4d1-…", "content_id": "b9a9116a-…", "status": "under_review",
 "logged_at": "2026-06-28T18:28:43.969Z", "message": "Appeal received. This content is now under review."}
```
See [planning.md](planning.md#4-appeals-workflow).

## Rate Limiting

Per client IP, via Flask-Limiter. Limits chosen for a realistic writing platform and to protect the
(cost-bearing) Groq calls from flooding:

| Endpoint | Limit | Why |
|---|---|---|
| `POST /submit` | **10/min, 100/hour, 300/day** | Each call costs a Groq request. A real creator submits a handful per session; 10/min absorbs honest re-check bursts, while hourly/daily caps blunt scripted flooding of the paid endpoint. |
| `POST /appeal` | **5/min, 20/hour** | Appeals are rare and human-initiated; low caps stop appeal-spam without blocking a legitimate creator. |
| `GET /log` | **30/min** | Read-only and cheap, but capped to discourage scraping the audit trail. |

Over-limit → HTTP `429` with a JSON error; the detection pipeline never runs.

**Evidence** — 12 rapid `POST /submit` requests against the 10/min limit (first 10 pass, rest blocked):

## Audit Log

Every `/submit` writes one structured row to a SQLite store (`provenance.db`, table `decisions`);
every `/appeal` writes an `appeals` row and flips the decision's `status`. Both print a one-line
console summary. `GET /log` returns the entries as JSON, newest first. Each row records: `content_id`,
`creator_id`, `title`, `timestamp` (UTC ISO `…Z`), `content_excerpt`, `attribution`, `confidence`,
`ai_likelihood`, `llm_score`, `stylometry_score`, `label_variant`/`label_text`, the full `signals`
array, `status`, and (if appealed) `appeal_id` / `appeal_reasoning` / `appealed_at`.

Real `GET /log` output — 3 structured entries spanning **all three verdicts**, one appealed
(`status: under_review` with `appeal_reasoning`). Trimmed for length; live runs include the full

## Known Limitations

These are honest failure modes tied to specific properties of the signals, not generic "needs more
data":

- **Non-native-English or formal/academic human writing → risk of a false "likely AI."** The LLM
  signal conflates a *uniform, formal register* with "AI-like," because that register is exactly what
  default LLM prose looks like. Since the LLM is the primary confidence driver, a confidently-wrong LLM
  here can push toward `likely_ai` *unless* stylometry actively dissents. This is the worst-case
  (false positive) for this system. Two things blunt it: the asymmetric AI bar (0.65) plus the
  stylometry-reliability cap often hold these to `uncertain` (see the academic example above), and the
  **appeal path exists precisely for the cases that slip through** — the appeal reasoning in the demo
  ("I am a non-native English speaker…") is this exact scenario.
- **Minimalist or repetitive poetry (haiku, spare free verse).** Stylometry reads the low
  sentence-length variance and limited vocabulary as "uniform → AI." Because stylometry can only
  *erode* confidence and never assert AI alone, and these texts trip the short-text reliability cap,
  the system lands on `uncertain` rather than accusing the poet — but it also means we genuinely
  cannot affirm human authorship for this content type.
- **Adversarial AI told to "vary sentence length and add a typo."** This defeats stylometry by
  construction and can soften the LLM's read, producing a false *negative* (missed AI). We accept this
  honestly: a missed AI is the tolerable error direction; a falsely-accused human is not.

## Spec Reflection

- **How the spec helped:** writing the *uncertainty representation* in `planning.md` first — the
  asymmetric threshold table and a concrete answer to "what does 0.6 mean to a user?" — gave the
  scoring code a precise target. When the AI-generated fusion came back, I could check it against that
  table line-by-line and immediately saw it diverged (it labeled obviously-AI text as `uncertain`).
  Without the written spec, that miscalibration would have looked "reasonable" and shipped silently.
- **How the implementation diverged, and why:** `planning.md` originally specified
  `confidence = decisiveness · (0.4 + 0.6·agreement)`. In testing (Milestone 4) this treated a *neutral*
  stylometry score as if it were *disagreement*, dragging confident, clearly-AI text down to
  `uncertain`. I diverged to an **LLM-primary** model (`base = 2·|LLM−0.5|`, stylometry corroborates or
  dissents) and updated the spec to match. The divergence actually *strengthened* the original thesis:
  it made detection work while keeping stylometry's power to veto a false positive, and it produced a
  cleaner fail-closed guarantee. (Minor divergence: field names — the spec used `submission_id` /
  `reasoning`; the implementation uses `content_id` / `creator_reasoning` to match the API contract.)

## AI Usage

I used an AI coding assistant throughout, directing it from the `planning.md` spec and reviewing every
output. Specific instances:

1. **Confidence fusion (directed → produced → overrode).** I gave it the detection-signals and
   uncertainty sections and asked it to generate `signal_stylometry` + the fusion logic. It produced a
   reasonable-looking weighted-average + `decisiveness·(0.4+0.6·agreement)` scorer. On testing, a
   clearly-AI paragraph (LLM 0.9) came back `uncertain` because the neutral stylometry score dragged
   confidence down. **I overrode the scoring model**, redesigning it to be LLM-primary, and re-verified
   against seven boundary pairs before keeping it.
2. **LLM signal robustness (directed → produced → strengthened).** I asked it to write the Groq call
   that returns a structured AI-likelihood. I revised the result to enforce a strict **fail-closed**
   path — JSON parse/validation failure returns a neutral `0.5` with `available: false` instead of
   raising or guessing — so an API error can never manufacture a confident "AI" accusation. I tested
   this branch explicitly (it correctly neutralized when the API was unreachable).
3. **Rate-limit error shape (directed → revised).** The generated Flask-Limiter setup returned the
   default HTML 429 page. I added a JSON error handler so `429` responses are machine-readable and
   consistent with the rest of the API.
