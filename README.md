# Provenance Guard

A backend service any creative-sharing platform can plug into to classify whether submitted
text was likely written by a human or generated by AI — then surface an honest, plain-language
transparency label and let creators appeal a classification.

> **Status:** Milestone 2 — full spec written. Implementation not started yet.
> Full design, diagrams, and the AI Tool Plan live in [planning.md](planning.md).

## Tech Stack

| Component          | Tool                                  |
|--------------------|---------------------------------------|
| API framework      | Flask                                 |
| Detection signal 1 | Groq (`llama-3.3-70b-versatile`)      |
| Detection signal 2 | Stylometric heuristics (pure Python)  |
| Rate limiting      | Flask-Limiter                         |
| Audit log          | SQLite                                |

## Architecture

A submission flows: `POST /submit` → **rate limiter** → **Signal 1 (LLM, Groq)** + **Signal 2
(stylometry)** → **confidence fusion** (weighting + agreement + asymmetric thresholds) → **label
builder** → **audit log (SQLite)** → JSON response. An appeal flows: `POST /appeal` → look up the
original decision → write a linked appeal row + set status `under_review` → response.

Full narrative and labeled diagrams for both flows: [planning.md](planning.md#architecture-diagram).

## Detection Signals

Two **distinct** signals — one semantic, one structural — so the combination beats either alone:

- **Signal 1 — LLM semantic classification (Groq `llama-3.3-70b-versatile`):** judges holistic
  voice/coherence (human vs AI). *Blind spot:* non-deterministic, weak on short text, and can
  flag bland-but-human writing — a false-positive risk.
- **Signal 2 — Stylometric heuristics (pure Python):** sentence-length variance ("burstiness"),
  type–token ratio, punctuation density. *Blind spot:* understands no meaning, unstable on short
  text, broken by poetry/lists, and easily gamed.

Their blind spots barely overlap, so disagreement is itself useful signal. See
[planning.md](planning.md#detection-signals).

## Confidence Scoring

We track `ai_likelihood ∈ [0,1]` (direction: 0 = human, 1 = AI) and a separate `confidence ∈ [0,1]`
(how strongly and consistently the evidence points). Fusion: `0.6·LLM + 0.4·stylometry`, then
`confidence = decisiveness · (0.4 + 0.6·agreement)` — so **signal disagreement suppresses confidence**.
Users never see the raw decimal; they see a confidence *band* + plain language.

The mapping is **not** a binary flip at 0.5 — it's an **asymmetric three-way** split (harder to assert
"AI" than "human", because false positives are the worst outcome):

| Condition | Verdict |
|---|---|
| `confidence < 0.50` | **Uncertain** |
| `confidence ≥ 0.50` & human side | **Likely human** |
| `0.50 ≤ confidence < 0.65` & AI side | **Uncertain** (not sure enough to accuse) |
| `confidence ≥ 0.65` & AI side | **Likely AI** |

So a confidence of **0.6 means "moderately sure"**: enough to state *likely human*, but on the AI side
it stays *uncertain*. Full formulas + calibration plan: [planning.md](planning.md#2-uncertainty-representation).

## Transparency Labels

The reader-facing label has three variants (final text below). Even the AI label is phrased as an
*estimate*, never an accusation, and names the appeal path.

| Variant                | Label text |
|------------------------|------------|
| High-confidence human  | ✍️ **Likely written by a person.** Our automated check found strong signs of human authorship. This is an automated estimate, not a guarantee. *(Confidence: High)* |
| High-confidence AI     | 🤖 **Likely AI-generated.** Our automated check found strong signs this was produced with AI assistance. This is an estimate, not a certainty — if you wrote this yourself, you can appeal. *(Confidence: High)* |
| Uncertain              | ❓ **Origin unclear.** Our automated check couldn't reliably tell whether a person or AI wrote this, so we're not making a claim either way. *(Confidence: Low)* |

## Appeals

A creator who disputes a classification calls `POST /appeal` with the `submission_id` and free-text
`reasoning`. The system looks up the original decision, logs a linked **appeal row**, flips the
decision's status to **`under_review`**, and returns confirmation. No automated re-classification —
appeals are flagged for a human reviewer, who sees the original text, verdict, signal scores,
confidence, label, and the creator's reasoning. See [planning.md](planning.md#4-appeals-workflow).

## Rate Limiting

Per client IP, via Flask-Limiter. Limits chosen for a realistic writing platform and to protect the
(cost-bearing) Groq calls from flooding:

| Endpoint | Limit | Why |
|---|---|---|
| `POST /submit` | **10/min, 100/hour, 300/day** | Each call costs a Groq request. A real creator submits a handful per session; 10/min absorbs honest re-check bursts, while hourly/daily caps blunt scripted flooding of the paid endpoint. |
| `POST /appeal` | **5/min, 20/hour** | Appeals are rare and human-initiated; low caps stop appeal-spam without blocking a legitimate creator. |
| `GET /log` | **30/min** | Read-only and cheap, but capped to discourage scraping the audit trail. |

Over-limit → HTTP `429`; the detection pipeline never runs. See [planning.md](planning.md#rate-limiting-limits-chosen--reasoning).
